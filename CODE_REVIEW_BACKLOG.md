## BL-001 · `wasApplied` in `dequeue()` always returned `false` · **auto-fixed**

| Field | Value |
|-------|-------|
| Area | `apps/web/src/hooks/useMessageQueue.ts` |
| Type | Bug |
| Impact | User-facing — auto-send message queue silently stopped working |
| Risk | Low (fix is self-contained) |
| Status | `auto-fixed` |

**Summary:** `wasApplied` was a closure variable mutated inside a `setQueue(updater)`
functional updater, then read as the return value immediately after `setQueue()` returned.
The author's own comment elsewhere in the same diff explicitly states that "React state
updater callbacks run asynchronously (during reconciliation)", meaning `wasApplied` would
always be `false` at return time. Both callers in `ChatView.tsx` checked `wasApplied &&
nextText`, so the auto-send queue was entirely disabled by this regression.

**Fix applied:** Removed the closure mutation inside the updater; the identity guard
(`latest[0]?.id !== firstItem.id`) still prevents double-dequeue at the state level.
`wasApplied: true` is now returned unconditionally when `queueRef.current` is non-empty,
consistent with the synchronous read that already verified an item exists.

## BL-002 · No file-upload size limit on `/api/transcribe`

| Field | Value |
|-------|-------|
| Area | `apps/server/src/http.ts` — `transcriptionRouteLayer` |
| Type | Bug / Performance |
| Impact | Internal / infra — unbounded memory use |
| Risk | Medium |
| Status | `auto-fixed` |

**Summary:** The new `POST /api/transcribe` multipart endpoint reads the uploaded file into
memory (`fileSystem.readFile(body.file.path)`) with no size cap. A client that uploads a
very large file (e.g. a multi-GB recording) can exhaust server RAM. The auth gate limits
exposure to authenticated clients but the risk still exists for self-hosted deployments.

**Fix applied:** Added `fs.stat(body.file.path)` before `readFile`; returns HTTP 413 if
size exceeds `FileSystem.MiB(50)`. Consistent with the existing Effect `FileSystem` service
pattern in the same handler.

## BL-004 · Hard page reload on dead WebSocket connection

| Field | Value |
|-------|-------|
| Area | `apps/web/src/wsRpcClient.ts` — `handleDeadConnection` |
| Type | UX / Design debt |
| Impact | User-facing — unsaved editor content lost on reconnect |
| Risk | Low |
| Status | `open` |

**Summary:** When the heartbeat declares the WebSocket connection dead, `window.location.reload()`
is called after a 50 ms delay. This is the simplest possible recovery strategy and avoids
stale subscription state, but it silently discards any unsaved file edits open in the
`FileEditorPane` (which tracks dirty state in Zustand/localStorage but not necessarily
flushed to disk). Users on poor connections (sleep/wake, VPN flap) will lose work.

**Suggested approach:** Before reloading, flush `filesPanelStore.dirtyByPath` to
`localStorage` so the editor pane can restore content on the next boot. Alternatively,
show a "Connection lost — reconnecting…" banner and attempt a clean WebSocket reconnect
before falling back to reload.

| Type | Bug (potential) |

| Risk | Low–Medium |
| Status | `resolved — safe as-is` |

**Summary:** `getComposerLineBlocks()` previously created an empty paragraph as a side

| Type | Bug (edge case) |
