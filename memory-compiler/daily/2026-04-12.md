### Session (20:01)

**Context:** Fixed a critical race condition in OllamaAdapter.ts where concurrent calls to `ensureOllamaRunning` could spawn multiple `ollama serve` processes simultaneously.

**Key Exchanges:**

- User identified race condition in `ensureOllamaRunning` (lines 196–231) where no serialization existed between concurrent callers and `spawn("ollama", ["serve"], ...)` calls
- Previous session's changes didn't persist to disk; needed full re-application in this session
- Clarified the serialization pattern: lock must use a piggyback mechanism where waiting callers re-check `pingOllama` themselves after awaiting the shared promise, rather than trusting the promise's resolved value directly

**Decisions Made:**

- Introduced module-level `ensureOllamaPromise: Promise<string | null> | null = null` as the serialization lock
- Fast-path: all callers first probe all candidate URLs directly before consulting the lock
- Piggyback path: if lock is held, concurrent caller awaits that promise, then calls `pingOllama` themselves to confirm and return the correct working URL
- Lock owner clears the promise in a `finally` block so future calls (e.g. user restarts Ollama) get a fresh spawn attempt
- **Rationale**: prevents multiple detached processes, ensures only first caller owns the spawn, subsequent callers share the result safely, and allows self-healing if Ollama crashes between messages

**Lessons Learned:**

- Session persistence issue: edits made in one session don't always persist to the next; always verify the true on-disk state before deciding fixes are already applied
- Race conditions in server-side helpers with external process spawning require both fast-path checks _and_ lock-guarded spawn paths to avoid thundering herd
- Piggyback callers must re-validate their own working URL after awaiting a shared promise (don't assume the promise's return value is their URL)

- Restart Bird Code server for changes to take effect (server-side Ollama integration fix)
