**Context:** Fixing broken "Send Now" button for queued messages in t3code chat application (multi-provider AI chat app)

**Key Exchanges:**

- User reported "Send Now" button doesn't work for queued messages across all providers
- Systematic debugging traced the issue through the full data flow from UI button → dequeue → send

**Decisions Made:**

- **Root cause identified as React 18 state updater timing trap**: `dequeue()` in `useMessageQueue.ts` used `setQueue(updaterFn)` and tried to read a ref set inside the updater, but React 18 runs updater functions asynchronously during reconciliation — so `lastDequeuedTextRef.current` was always `null` when `dequeue()` returned
- **Fix approach**: Replaced async-dependent pattern with a `queueRef` that mirrors committed queue state for synchronous reads, plus identity guard in functional updater to prevent double-dequeue
- Both `onSendQueuedNow` (manual send) and auto-send effect (after AI finishes) were broken by the same root cause

**Lessons Learned:**

- **React 18 `setState(updaterFn)` timing trap**: The updater function passed to `setState` runs during reconciliation, NOT synchronously. You cannot set a ref inside an updater and read it after `setState` returns — it will still have its pre-call value. This is a subtle but critical distinction from class component `setState` callbacks.
- Pattern to avoid: `setX(fn => { ref.current = value; return newState; }); return ref.current; // ← always stale`
- Correct pattern: Use a ref that mirrors committed state (`queueRef.current = queue` on every render) for synchronous reads, then schedule state updates separately
