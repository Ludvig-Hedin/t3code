### Session (23:04)

**Context:** Debugging frozen "working" state and broken stop button in a multi-provider chat app (ClaudeAdapter / CodexAdapter), plus sidebar spinner/stop-button flicker.

---

**Key Exchanges:**

- User reported two bugs: (1) AI freezes and shows "working" indefinitely, (2) stopping a thread still shows the spinner in sidebar and briefly flips stopâ†’defaultâ†’stop button again.
- Root cause investigation revealed: `runSdkStream` blocks forever when Claude CLI subprocess freezes (no inactivity watchdog); `query.interrupt()` has no timeout so stop is a no-op if the subprocess is frozen; client `derivePhase` shows `running` purely from session status without requiring an active turn ID.

---

**Decisions Made:**

- Add a 5s timeout to `query.interrupt()` in ClaudeAdapter and CodexAdapter; on timeout, fall through to `stopSessionInternal` (force fiber interrupt). Used `Effect.timeoutOption` to avoid adding error types.
- Add an inactivity watchdog fiber inside `runSdkStream`: reset `lastActivityAtMs` on every SDK message; if no activity for configurable N seconds while a turn is in flight, call `stopSessionInternal("failed", "Provider stream timed out")`.
- Track `lastActivityAtMs` via fiber ref so watchdog and stream fiber share state safely.
- Add late-event ingestion guard in `handleStreamExit`/event pipeline: ignore terminal events for sessions with status `"stopped"` or `"completed"` to prevent zombie events re-activating the UI.
- Client `derivePhase`: require `activeTurnId` to be set in addition to `session.status === "running"` before returning `"running"` phase â€” prevents spinner from lingering after turn is gone.

---

**Lessons Learned:**

- `Stream.takeWhile(() => !context.stopped)` is useless as a stop guard if no messages ever arrive â€” the predicate never evaluates. A watchdog fiber is the correct pattern for freeze detection.
- Effect's `Effect.timeout` adds `TimeoutException` to the error channel; use `Effect.timeoutOption` instead to keep the error type clean.
- Pre-existing type errors were already present in the codebase; new changes passed typecheck cleanly.
